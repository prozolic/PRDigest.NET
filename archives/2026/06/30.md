### [#128979](https://github.com/dotnet/runtime/pull/128979) Escape quotes and backslashes in MailAddress display name when encoding SMTP headers {#128979}
- 作成者: [@Copilot](https://github.com/apps/copilot-swe-agent)
- 作成日時: 2026年06月04日 11:51:23(UTC)
- マージ日時: 2026年06月30日 11:05:13(UTC)
- ラベル: <a style="text-decoration:none;" href="../../labels/area-System.Net/index.html"><span style="background-color: #d4c5f9; color: #000000; display: inline-block; padding: 0 7px; font-size:12px; font-weight:500; line-height:18px; border-radius:2em; border:1px solid transparent;">area-System.Net</span></a>
#### 概要
`MailAddress.Encode` で生成される SMTP ヘッダーの表示名（DisplayName）に含まれるダブルクォート（`"`）とバックスラッシュ（`\`）を RFC 5322 準拠のように正しくエスケープする修正です。`Henry "The Fonz" Winkler` や `C:\path\to\file` のような特殊文字を含む表示名を使用した際に、不正な quoted-string が生成される問題を解決します。

#### 変更内容
- **MailAddress.Encode (ASCII / allowUnicode=true ブランチ):** バックスラッシュを `\\` に、ダブルクォートを `\"` にエスケープし、RFC 5322 準拠の quoted-string を生成するよう修正
- **MailAddress.ToString():** `\` → `\\` のエスケープを追加し、Encode パスとの一貫性を確保
- **エスケープ順序:** 両呼び出し側でバックスラッシュを先にエスケープした後にダブルクォートをエスケープ（順序を逆にすると二重エスケープが発生）
- **テスト拡充:** MailAddressEncodeTest に回帰テスト（DisplayNamesWithSpecialChars データソース）を追加し、クォートのみ、バックスラッシュのみ、両者の組み合わせケースをカバー；MailAddressDisplayNameTest のテスト データを拡張しバックスラッシュ変種を追加

#### パフォーマンスへの影響
影響なし

#### 関連Issue
[#36752](https://github.com/dotnet/runtime/issues/36752)（クォート対応の当初実装）

#### その他
Q エンコード（非 ASCII）ブランチは影響を受けません（encoded-word 構文は quoted-string 構文を使用しないため）。この修正により、特殊文字を含む表示名でも正しい SMTP ヘッダーが生成されるようになります。

### [#128862](https://github.com/dotnet/runtime/pull/128862) Add Process.Start callback overload with WindowsProcessStartArguments and UnixProcessStartArguments {#128862}
- 作成者: [@Copilot](https://github.com/apps/copilot-swe-agent)
- 作成日時: 2026年06月01日 17:35:46(UTC)
- マージ日時: 2026年06月30日 12:25:24(UTC)
- ラベル: <a style="text-decoration:none;" href="../../labels/area-System.Diagnostics.Process/index.html"><span style="background-color: #d4c5f9; color: #000000; display: inline-block; padding: 0 7px; font-size:12px; font-weight:500; line-height:18px; border-radius:2em; border:1px solid transparent;">area-System.Diagnostics.Process</span></a>
#### 概要

`Process.Start()` にコールバックベースのオーバーロードを追加し、開発者がカスタムプラットフォームAPI（Windows の `CreateProcess` や Unix の `posix_spawn` など）でプロセスを作成しながら、.NET の `Process` インフラストラクチャ（ストリーム処理、`WaitForExit` など）を活用できるようにしました。新しい `ref struct` 型 `WindowsProcessStartArguments` と `UnixProcessStartArguments` は、プリペアされたスタート情報（標準ハンドル、コマンドラインやargv、環境ブロック、Unix では UTF-8 ポインタとしての実行可能ファイルパス）をコールバックに渡します。

#### 変更内容

- **Process.Start() オーバーロード追加**（公開API）: `Process.Start(ProcessStartInfo, Func<WindowsProcessStartArguments, IntPtr>)` と `Process.Start(ProcessStartInfo, Func<UnixProcessStartArguments, int>)` を追加。Windows はプロセスハンドルを、Unix はプロセスIDを返します。
- **新しい ref struct 型**（公開API、`System.Runtime.InteropServices` 名前空間）: `WindowsProcessStartArguments`（標準ハンドル、コマンドラインなど）と `UnixProcessStartArguments`（argv/envp、解決されたパスへの `byte*` ポインタなど）
- **SafeProcessHandle へのコールバック支援**: Windows で環境ブロック＋コマンドラインの構築・ピニング、Unix で argv/envp の割り当てと待機状態登録を実装
- **ProcessUtils.Unix の統一パス検証**: `ProcessUtils.ResolveValidPath()` が空白のみのワーキングディレクトリを `null` に正規化
- **トリミング対応**: `StartCore` と `StartCoreWithCallback` を完全に分離し、`SafeProcessHandle.StartWithCallback` はコールバックAPI を明示的に使用するアプリケーションにのみ含まれるようにしました
- テスト追加: Windows・Unix での stdout/stderr リダイレクション検証用コールバックテストを実装

#### パフォーマンスへの影響

影響なし。コールバックベースの呼び出しパスと従来のパスは分離されており、既存コードのパフォーマンスに変化はありません。

#### 関連Issue

[#71595](https://github.com/dotnet/runtime/issues/71595)

#### その他

- **破壊的変更なし**: 新しいオーバーロード追加のみで、既存API は変更なし
- **プラットフォーム標注**: `[SupportedOSPlatform]`/`[UnsupportedOSPlatform]` はメソッドレベルに配置
- **セキュリティ**: `ref struct` 制約によりポインタ型プロパティの生存期間はコールバック呼び出しに限定
- **API ガイダンス**: 元の `ProcessStartArguments.FileName` は廃止され、Unix 専用の `UnixProcessStartArguments.ResolvedPath`（`byte*`）に置き換わりました
- **リスク判定**: 中程度。新しい公開API サーフェスと参照アセンブリ整合性の検証が必要ですが、コールバック固有パスに限定した変更により軽減されています
