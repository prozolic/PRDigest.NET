#### 変更内容
- **JIT コンパイラ（x64/ARM64）**: HWIntrinsic 命令リストに新しいレーン構築・合成操作を追加し、それぞれのプラットフォーム固有の実装（hwintrinsicxarch.cpp, hwintrinsicarm64.cpp）を実装

- 作成者: [@lewing](https://github.com/lewing)
- 作成日時: 2026年06月18日 19:12:56(UTC)
- マージ日時: 2026年06月19日 06:38:08(UTC)
- ラベル: <a style="text-decoration:none;" href="../../labels/area-Build-mono/index.html"><span style="background-color: #d4c5f9; color: #000000; display: inline-block; padding: 0 7px; font-size:12px; font-weight:500; line-height:18px; border-radius:2em; border:1px solid transparent;">area-Build-mono</span></a>
#### 概要
Wasm.Build.Tests の `ToolCommand.ExecuteWithCapturedOutput` でプロセスの `Exited` イベントと非同期出力ハンドラ間の競合状態を修正します。高速に終了するサブプロセス（例：Hello-World の `dotnet run`）で、出力ハンドラがすべてのデータを処理する前にプロセスが終了すると、キャプチャされた出力が空になる間欠的な失敗が発生していました。修正では `await completionTask` 後に `Process.WaitForExit()` を追加し、最終的な出力ドレイン（stdout/stderr の完全なフラッシュ）を強制します。

#### 変更内容
- **ProcessExtensions.cs**: 不要になった `StartAndWaitForExitAsync` メソッドを削除（-55行）
- **ToolCommand.cs**: `ExecuteWithCapturedOutput` メソッドを修正し、プロセス完了タスク待機後に同期 `WaitForExit()` を呼び出すことで非同期出力ハンドラの完全なドレインを保証（+31/-38行）、競合状態を回避するためのコメントを追加

#### パフォーマンスへの影響
`WaitForExit()` はテスト実行時のみ呼び出されるため、製品コードへの影響はありません。テスト実行時は追加の同期待機が発生しますが、これはプロセス終了後の出力バッファドレイン完了を待つもので、通常は数ミリ秒以内に完了します。

#### 関連Issue
なし

このプルリクエストは GitHub Copilot の支援で作成されたものです。修正は Helix ワーカーで約12個の xunit theories が並行実行されている環境での間欠的な失敗に対応しています。

- **ISOSDacInterface17 定義**：`SOSStressLogData`、`SOSThreadStressLogData`、`SOSStressMsgData` データ構造体、`ISOSStressLogThreadEnum`/`ISOSStressLogMsgEnum` 列挙子インターフェース、および `GetStressLogData`、`GetStressLogThreadEnumerator`、`GetStressLogMessageEnumerator` メソッドを IDL で定義

- **ストレスログが有効でない場合**、すべての API は `S_FALSE` を返す
- **レガシー DAC は非対応**：ネイティブ DAC はこのインターフェースを実装しておらず、cDAC のみが対応

#### パフォーマンスへの影響
影響なし。既存の cDAC `IStressLog` コントラクトの上に構築された COM ブリッジであり、ランタイムのパフォーマンス特性に変更なし。

#### 概要

Vector型に対する**レーン構築・合成API**と**シーケンス生成API**を追加するPRです。幾何級数、交互シーケンス、調和級数の生成、および`Concat/Zip/Unzip/Reverse`などのレーン操作機能が新たに公開APIとなります。JITはこれらをネイティブSIMD命令に展開し、ハードウェア非対応時は狭いベクトル幅での分解をサポートします。

#### 変更内容

**公開API追加（ref assemblies）:**
- `Vector<T>/Vector{64,128,256,512}<T>`に`CreateGeometricSequence`、`CreateAlternatingSequence`、`CreateHarmonicSequence`を追加
- `Concat*`（Lower/Upper組合せ）、`Zip/ZipLower/ZipUpper`、`Unzip/UnzipEven/UnzipOdd`、`Reverse`レーン操作を追加
- `SignSequence`を符号付き数値型向けの拡張プロパティとして公開
- `CreateCauchySequence`は削除

**JIT実装（x86-64ターゲット）:**
- 固定イミディエイトシャッフル（`vpbroadcast*`、`vshufps`、`vpunpck*`）により128/256ビットのconcat/zip/unzipを最適化
- AVX512時は512ビット対応（`vpermt2d`、`vshufi32x4`等）
- ハードウェア非対応時はVector256への分解機構
- 定数入力時は完全に計算済みベクトルをリード・オンリーデータに埋め込み

**JIT実装（ARM64ターゲット）:**
- テーブルルックアップを避け、`ins`（要素挿入）、`zip1/zip2`、`uzp1/uzp2`、`rev64`等の直接命令を活用

**マネージド実装（CoreLib）:**
- Vector64～Vector512に対する完全な実装
- 部分ベクトル幅（`simdCount==1`）の特殊ケース処理

**テスト追加:**
- Vector64/128/256/512およびSystem.Numerics.Vectorに対する包括的なユニットテスト

#### パフォーマンスへの影響

**改善:**
- 128ビットconcat: 3命令（`vmovups`/`vshufps`/`vmovups`）に最適化
- Reverse: 固定シャッフルイミディエイト1命令（例：`vpshufd xmm0, [rdx], 27`）
- 定数入力時：シーケンス計算はJIT時実行済み、ランタイムはメモリロードのみ（約19バイト）
- Vector256/512への分解：AVX512未対応環境でも効率的に動作

**懸念点（レビューコメント）:**
- `CreateGeometricSequence`の浮動小数点版で丸め結果が異なる可能性：各レーン`initial * (multiplier^index)`として計算されるため、逐次乗算による積とは異なる
- `elementCount`変数が未使用で警告の可能性あり
- `CreateAlternatingSequence`のAVX2高速パスで64ビット広報による型ミスマッチ（`TYP_LONG`で作成されるが周辺は`TYP_INT`）
