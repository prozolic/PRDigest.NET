### [#129950](https://github.com/dotnet/runtime/pull/129950) Emit correct ELF symbol types for data in NativeAOT objects {#129950}
- 作成者: [@max-charlamb](https://github.com/max-charlamb)
- 作成日時: 2026年06月28日 18:56:57(UTC)
- マージ日時: 2026年06月28日 23:38:02(UTC)
- ラベル: <a style="text-decoration:none;" href="../../labels/area-NativeAOT-coreclr/index.html"><span style="background-color: #d4c5f9; color: #000000; display: inline-block; padding: 0 7px; font-size:12px; font-weight:500; line-height:18px; border-radius:2em; border:1px solid transparent;">area-NativeAOT-coreclr</span></a>
#### 概要
NativeAOT の ELF オブジェクトライター内で、シンボルテーブル生成時のシンボル型分類ロジックを修正しました。従来はシンボルサイズのみで型を決定していたため、非実行可能セクション内のデータシンボルが誤った型（`STT_NOTYPE` または `STT_FUNC`）として出力されていました。本変更はセクションフラグに基づいて正しい型を割り当て、デバッガ（lldb/gdb/nm など）での正確なシンボル認識を実現します。

#### 変更内容
- **src/coreclr/tools/Common/Compiler/ObjectWriter/ElfObjectWriter.cs**
  - `EmitSymbolTable` メソッドのシンボル型分類ロジックを修正
  - TLS セクション: `STT_TLS`、非実行可能セクション: `STT_OBJECT`、実行可能セクション: `STT_FUNC`/`STT_NOTYPE` で分類
  - 修正前：シンボルサイズ（`Size > 0`）のみで判定し、vtable が `STT_NOTYPE`、`.rodata` のデータが `STT_FUNC` として誤分類
  - 修正後：セクションフラグを優先的に参照し、実行可能セクション内のシンボル（コード）は既存動作を維持

#### パフォーマンスへの影響
影響なし。本変更は `st_info` 型バイトのみを変更するもので、オブジェクトファイルのサイズは変わりません（検証：修正前後で 3,555,456 バイト同一）。

#### 関連Issue
なし

#### その他
- **検証結果**（Linux x64 cross-emit、データ脱水化有効時）：
  - `.hydrated` セクション：1084 個の `NOTYPE` + 377 個の `FUNC` → 1461 個の `OBJECT` に統一
  - `.rodata` セクション：245 個の `FUNC` + 2 個の `NOTYPE` → 247 個の `OBJECT` に統一
  - `.data` セクション：338 個の `FUNC` → 338 個の `OBJECT` に統一
  - `.text` セクション（コード）：83 個の `FUNC` のまま（変更なし）
- 実行可能セクション内のシンボルは変更されないため、ARM/AArch64/RISC-V のマッピングシンボル（`$a`/`$d`/`$x`）とトランポリンに影響なし
- 他の ELF プロデューサ（gcc/llvm など）の標準的な型分類規則に準拠

- 古いバージョンの DAC で生成されたダンプには該当ページが含まれないため、この修正は新規作成されるダンプに適用されます

#### 概要
シンロック（オブジェクトヘッダ）の取得/解放処理をCoreClrでネイティブコードからマネージドコードに移行し、CoreClrとNativeAOT間の実装の共有を拡大させました。また、シンロックが他のスレッドに所有されている場合にTryAcquireが過度にスピンする問題を修正しました。パフォーマンスはプラットフォームとロック方式に応じて最大+30%の改善を実現しており、一部で軽微な-3%程度の回帰が見られますが、これはハードウェア差による「アーキテクチャノイズ」と評価されています。

#### 変更内容
- **ネイティブコード削除**: `syncblk.inl`, `syncblk.h`のシンロック実装、`comsynchronizable.cpp/.h`のFCALL実装、`ecalllist.h`のFCALL/ecall マッピングを削除
- **マネージドコード実装（CoreCLR）**: `ObjectHeader.CoreCLR.cs`にシンロック取得/解放ロジックを実装し、`AcquireThinLock()`と`Release()`を公開
- **Monitor統合**: `Monitor.CoreCLR.cs`から157行削除し、新しいマネージドシンロックエントリポイントへのルーティングに変更
- **NativeAOT同期**: `ObjectHeader.NativeAot.cs`と`Monitor.NativeAot.cs`を更新し、統一された`AcquireThinLock()`エントリポイントに対応
- **ビルド設定**: CMakeLists.txt、common.h、pal.hから不要なincludesと参照を削除

#### パフォーマンスへの影響
- **改善**: スループットで最大+30%の向上（プラットフォームとシンロック/ファットロック方式に依存）
- **軽微な回帰**: 一部シナリオで最大-3%
- **評価**: 小さな差異はコード生成やハードウェア差異（AMD Epyc 7763 vs AMD Ryzen 7950など）に由来する「アーキテクチャノイズ」と判断
