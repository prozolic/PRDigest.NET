従来はMethodImplが存在する場合に全ての仮想スロットの埋め込みをスキップしていたため、O(V*N)の二次計算量が発生していました（V：仮想スロット数、N：型階層内のメソッド数）。

本変更では、正規MethodTableのvtableで`NULL`エントリのスロットを検出し、これらのスロットについては親チェーンをたどって直接埋め込むことで、WPF等の深い階層を持つ型の起動時パフォーマンスを改善します。

#### 変更内容
- **src/coreclr/vm/methodtable.h**: `GetSlotForVirtualVolatileLoadWithoutBarrier` ヘルパー追加（正規vtableスロットをメモリバリアなしで読み取り）
- **src/coreclr/vm/methodtable.cpp**: `FillEntryDataForAncestor` 実装を変更
  - MethodImpl存在時に正規vtable内の`NULL`スロットを検出
  - `NULL`スロット（上書きされていない仮想メソッド）の場合、親チェーンをたどって導入型までDecl/Impl MethodDescを埋め込み
  - 一般的なMethodImplケースの保守的な早期リターンは維持

**改善あり（WPF起動時性能）:**

- App2（メニュー+ツールバー+50行DataGrid）: **1455.9 ms → 1377.9 ms（−5.4%、−78.0 ms）**

測定は正規MethodTable読み込み処理の型ロード時間削減に特化しており、MethodImplが実際に存在する階層でのみ追加処理が実行されるため、一般的なケースへの影響はありません。

- 作成者: [@Copilot](https://github.com/apps/copilot-swe-agent)
- 作成日時: 2026年05月22日 22:40:52(UTC)

#### 概要
.NET 8+ で `DataContractSerializer` が `Nullable<T>` の `[DataContract]` 構造体を逆シリアル化する際にキャッシュルックアップループのバグにより `SerializationException`（DataContract cache overflow）が発生する回帰を修正しました。キャッシュスキャンループが元の計算済みIDと比較していたため、有効なコントラクトが見つからずエラーが発生していました。

修正内容：
```csharp
// 修正前：同じidと繰り返し比較
for (int i = 0; i < currentDataContractId; i++)
{
    if (ContractMatches(classContract, s_dataContractCache.GetItem(id)))  // idを繰り返し使用
    {
        return i;
    }
}

// 修正後：ループインデックスiで正しくスキャン
for (int i = 0; i < currentDataContractId; i++)
{
    if (ContractMatches(classContract, s_dataContractCache.GetItem(i)))  // iを使用
    {
        return i;
    }
}
```

#### 変更内容
- **System.Private.DataContractSerialization**: `DataContract.GetIdForInitialization(ClassDataContract)` でキャッシュスキャンループをループ変数 `i` でインデックスするよう修正
- **System.Runtime.Serialization.Xml テスト**: `DCS_NullableDataContractStructAsRoot_DeserializesWithoutPriorInitialization` という回帰テストを追加。事前のキャッシュ初期化がない状態で nullable root `[DataContract]` 構造体を `ReadObject` で逆シリアル化するテスト

#### パフォーマンスへの影響
影響なし。バグ修正であり、パフォーマンスに関連する変更ではありません。
