- 作成者: [@tannergooding](https://github.com/tannergooding)
- 作成日時: 2026年07月21日 03:20:03(UTC)
- マージ日時: 2026年07月21日 18:44:29(UTC)
- ラベル: <a style="text-decoration:none;" href="../../labels/area-System.Numerics/index.html"><span style="background-color: #d4c5f9; color: #000000; display: inline-block; padding: 0 7px; font-size:12px; font-weight:500; line-height:18px; border-radius:2em; border:1px solid transparent;">area-System.Numerics</span></a> <a style="text-decoration:none;" href="../../labels/breaking-change/index.html"><span style="background-color: #ffa500; color: #000000; display: inline-block; padding: 0 7px; font-size:12px; font-weight:500; line-height:18px; border-radius:2em; border:1px solid transparent;">breaking-change</span></a>
#### 概要
`Complex<T>` および非ジェネリック `Complex` を C23 Annex G（IEC 60559互換複素算術）の特殊値（符号付きゼロ、無限大、NaN）要件に準拠させます。乗算・除算時に無限大回復メカニズムを導入し、初等関数（`Sqrt`、`Exp`、`Log`など）に Annex G プロローグを追加します。これは特殊値入力の動作を変更する**破壊的変更**であり、以前は NaN を返していた多くのケースで Annex G が要求する有向無限大、符号付きゼロ、または特定の符号付き結果を返すようになります。

#### 変更内容
- **Complex.Generic.cs**: `operator *` に Annex G.5.1 無限大回復を実装し、`operator /` に Smith の公式と回復機構を層状に適用。初等関数に Annex G プロローグを追加。スカラー乗除算やその他の演算は複素演算パスを経由して特殊値動作の一貫性を確保
- **Complex.cs**: 非ジェネリック `Complex` の演算と数学関数を `Complex<double>` に委譲して更新された特殊値動作を継承
- **ComplexTests.SpecialValues.cs**: `Complex<double/float/Half>` 向けの新規汎用特殊値テストハーネスを追加。独立した Annex G リファレンスから期待値を生成
- **ComplexTests.cs**: 既存テストのオラクルと期待値を Annex G 特殊値動作に合わせて更新
- **System.Runtime.Numerics.Tests.csproj**: 新テストファイルをプロジェクトに追加

#### パフォーマンスへの影響
`operator /(Complex<T>, T)` がスカラー除数を複素値に昇格させて複素演算パスを経由するため、有限かつゼロでない除数の一般的なケースで追加の処理コストが発生する可能性があります。

#### 関連Issue
[#131132](https://github.com/dotnet/runtime/issues/131132)
破壊的変更ドキュメント: [dotnet/docs#54845](https://github.com/dotnet/docs/issues/54845)

#### その他
- 全 System.Runtime.Numerics テストスイート: 8368 合格、0 失敗、0 スキップ
- 本 PR は GitHub Copilot の支援を受けて起案されました
- `operator /` は Smith の公式を保持したまま回復を重ね、Annex G リファレンスアルゴリズムがオーバーフロー可能な大きさの被除数に対しても正確性を維持する設計

- 作成者: [@max-charlamb](https://github.com/max-charlamb)

#### 概要
cDAC（コンポーネント診断契約）の`TypeHandle`値型を`ITypeHandle`インターフェースに変換し、ターゲット指向の実装（`TargetTypeHandle`）を追加するリファクタリングです。これにより、ランタイム型システム（RTS）が型ハンドルのバージョニングと合成型の生成に対応できるようになります。主な特徴は、ハンドルの比較をアドレス値から正規参照ID（canonical reference identity）に変更し、`Target.Flush`までの間は同じアドレスに対して同一オブジェクトを返します。

#### 変更内容
- **ITypeHandleインターフェース導入**：抽象型（Abstractions）に`ITypeHandle`を追加し、既存コードの`TypeHandle`参照をすべて更新
- **TargetTypeHandle実装**：ターゲットバックの実装クラスを追加し、アドレスごとのインターンでキャッシュを最適化
- **Nullable参照セマンティクス**：不在・未解決型を`ITypeHandle?`で表現
- **契約署名の更新**：RuntimeTypeSystem、RuntimeMutableTypeSystem、Signature、ManagedTypeSourceなどの公開契約をすべて`ITypeHandle`に対応
- **ジェネリック型インスタンスをArray返却に変更**：`GetGenericParameters`などが`ITypeHandle[]`を返すように統一
- **ドキュメント更新**：RuntimeTypeSystem.md含む契約ドキュメントを更新
- **テスト・レガシーコード更新**：2733個のcDAC単体テスト、34個のDataGenerator テスト、DumpTests/StressTests、SOS DAC実装（Legacy）をすべて対応

#### パフォーマンスへの影響
- ハンドルインターンによるメモリ最適化：キャッシュ有効期間内は重複ハンドルの生成を排除
- 参照ID比較（`ReferenceEquals`）がアドレス値比較より高速な可能性あり
- キャッシュ無効化（`Target.Flush`）後は新規オブジェクト生成のため初期オーバーヘッド発生
- 具体的なベンチマーク数値は提供されていません

#### 関連Issue
なし

#### その他

- **合成型ハンドルの延期**：Ptr/Byref/SzArray/Array、ジェネリック型のような無限個の構築型に対する合成ハンドル生成は別のPRで実装予定
