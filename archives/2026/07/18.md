### [#131000](https://github.com/dotnet/runtime/pull/131000) Implement wasm codegen for sub-16 SIMD load/store {#131000}
- 作成者: [@tannergooding](https://github.com/tannergooding)
- 作成日時: 2026年07月17日 21:45:17(UTC)
- マージ日時: 2026年07月18日 14:20:38(UTC)
- ラベル: <a style="text-decoration:none;" href="../../labels/area-CodeGen-coreclr/index.html"><span style="background-color: #d4c5f9; color: #000000; display: inline-block; padding: 0 7px; font-size:12px; font-weight:500; line-height:18px; border-radius:2em; border:1px solid transparent;">area-CodeGen-coreclr</span></a>
#### 概要
WebAssembly JITコンパイラの`Vector2`（`TYP_SIMD8`）と`Vector3`（`TYP_SIMD12`）に対するSIMDメモリ操作（load/store）のコード生成を実装しました。これらの型はネイティブなwasm valtypeを持たないため、`v128`として8/12バイトのみを使用しており、従来はNYI（Not Yet Implemented）でベイルアウトしていました。レーン単位のload/store命令シーケンスを発行することで、この制限を回避します。

**発行される命令シーケンス:**
- `simd8 load`: `v128.load64_zero 0`
- `simd8 store`: `v128.store64_lane 0, lane 0`
- `simd12 load`: `local.get addr; v128.load64_zero 0; v128.load32_lane 8, lane 2`
- `simd12 store`: 一時的な`v128`に値をコピー、低8バイトを`v128.store64_lane`、上位4バイトを`v128.store32_lane`で処理

#### 変更内容
- **codegenwasm.cpp**: `TYP_SIMD8`と`TYP_SIMD12`に対する分割レーンload/storeシーケンスを実装、アドレス再生成を含むヘルパールーチンを追加
- **instr.cpp**: `ins_Load(TYP_SIMD8)`を`INS_v128_load64_zero`として実装
- **lowerwasm.cpp**: コード生成がアドレスを再利用/再生成する必要がある`TYP_SIMD12`間接参照アドレスに対して、複数使用をマーク
- **regallocwasm.cpp**: `TYP_SIMD12` store間接参照（書き換えで導入されたSTOREINDを含む）に対して内部`v128`レジスタを割り当て

#### パフォーマンスへの影響
corelib crossgen2ブラウザSuperPMIコレクション（27,540コンテキスト）での測定結果：
- ハードアサーション: **403 → 30** に削減
- **373個**の`SIMD8`/`SIMD12` load/store アサーション排除
- パフォーマンス低下なし
- 残存する30個のアサーションは既存の`NYIRAW`汎用キャッチオール（本変更と無関係）

#### 関連Issue
[#130866](https://github.com/dotnet/runtime/issues/130866)（SIMD ABI ベイルアウトのシャドウイング除去）

#### その他
この変更は単体でもcorelib crossgen2コレクションの373個のアサーションを解決しますが、完全な効果を得るには#130866の適用が必要です。ローカル・スタック・ストア書き換え処理（`RewriteLocalStackStore`）で生成される合成STOREIND内の`v128` teeレジスタは主要な収集ウォーク外のため、`RewriteLocalStackStore`内で明示的に要求されます。

#### 概要
WebAssembly バックエンド向けに `System.Runtime.Intrinsics.Wasm.WasmBase` の `LeadingZeroCount`/`TrailingZeroCount` 組み込み関数を JIT で実装します。これまで未実装のため管理コードの低速フォールバック実装を使用していましたが、本変更により RyuJIT-wasm が native wasm の `clz`(count leading zeros)/`ctz`(count trailing zeros) 命令へ直接降級できるようになります。

#### 変更内容
- **hwintrinsic.cpp** — WasmBase の IsaRange を `{ NI_Illegal, NI_Illegal }` から `{ FIRST_NI_WasmBase, LAST_NI_WasmBase }` に修正し、実装対象として認識させる
- **hwintrinsiclistwasm.h** — スカラー版の `LeadingZeroCount`/`TrailingZeroCount` (int/uint/long/ulong) を `HW_Category_Scalar` で追加、`HW_Flag_SpecialImport|HW_Flag_NoFloatingPointUsed` フラグを指定して既存の xarch パターンに合わせる
- **hwintrinsicwasm.cpp** — 両関数を `NI_PRIMITIVE_LeadingZeroCount`/`NI_PRIMITIVE_TrailingZeroCount` に special-import してから `GT_INTRINSIC` ノードに変換（既存の wasm コード生成が `i32/i64.clz`/`ctz` に降級可能）

#### パフォーマンスへの影響
`System.Numerics.BitOperations` の `LeadingZeroCount`/`Log2`/`TrailingZeroCount` が WasmBase で利用可能な場合、管理コードのソフトウェア実装から native wasm `clz`/`ctz` 命令へ置き換わるため、パフォーマンスが向上します。具体的な数値は記載されていません。

#### 関連Issue
なし

#### その他
- **内部実装のみの変更** — `WasmBase` は `internal` クラスで既に main ブランチに存在しており、本 PR は JIT 降級の実装に過ぎません。公開 API への変更はありません。
- **検証状況** — JIT コンパイルは成功（警告・エラーなし）し、降級対象の `NI_PRIMITIVE_*ZeroCount` は既に `codegenwasm.cpp` で処理されているため、既証のコード生成パスを再利用しています。実運用環境でのテスト実行は試みられていません（WebAssembly-CoreCLR 実験版ランタイムの bring-up 問題により）。既存の `System.Numerics.BitOperations` テストが全オーバーロードをカバーしており、ランタイムの完成後にこのコードパスが演習されます。

### [#130924](https://github.com/dotnet/runtime/pull/130924) \[wasm\] Publish __stack_pointer before SuppressGCTransition native calls {#130924}
- 作成者: [@lewing](https://github.com/lewing)
- 作成日時: 2026年07月16日 21:13:37(UTC)
- マージ日時: 2026年07月18日 00:12:50(UTC)
- ラベル: <a style="text-decoration:none;" href="../../labels/arch-wasm/index.html"><span style="background-color: #eb6420; color: #ffffff; display: inline-block; padding: 0 7px; font-size:12px; font-weight:500; line-height:18px; border-radius:2em; border:1px solid transparent;">arch-wasm</span></a> <a style="text-decoration:none;" href="../../labels/area-CodeGen-coreclr/index.html"><span style="background-color: #d4c5f9; color: #000000; display: inline-block; padding: 0 7px; font-size:12px; font-weight:500; line-height:18px; border-radius:2em; border:1px solid transparent;">area-CodeGen-coreclr</span></a>
#### 概要
WebAssembly R2R（Ready-to-Run）において、`SuppressGCTransition`ネイティブ呼び出し前に shadow stack pointer を `__stack_pointer` グローバル変数に発行する機能を追加します。これにより、emscripten などの `__stack_pointer` に依存するネイティブコードが正しいスタック位置を観測し、R2R フレーム上のローカル変数破損を防ぎます。

```csharp
// 修正前: __stack_pointer がstaleのままnative codeが実行される
// → native calleeが呼び出し元のSP（R2Rフレーム上）からシャドウフレームを割り当て
// → spilled pointer が破損 → 後続のメモリアクセスでNullReferenceException

// 修正後: SuppressGCTransition呼び出し直前に shadow SP を発行
local.get <shadowSP>
global.set <__stack_pointer>
// → native calleeが正しく呼び出し元フレーム下にシャドウフレームを割り当て
```

#### 変更内容
- **src/coreclr/jit/codegenwasm.cpp**: `IsUnmanaged() && IsSuppressGCTransition()` 呼び出し直前に、shadow SP local を `__stack_pointer` グローバル（`eeGetWasmWellKnownGlobals()->stackPointer` から取得）に発行するコードを追加（+14行）

#### パフォーマンスへの影響
wasm operand stack上で net-zero であるため、パフォーマンス低下はなし。

#### 関連Issue
[#130923](https://github.com/dotnet/runtime/issues/130923)、[#130924](https://github.com/dotnet/runtime/issues/130924)（本PR）
依存: [#129717](https://github.com/dotnet/runtime/issues/129717)、関連: [#130634](https://github.com/dotnet/runtime/issues/130634)

#### その他
- ドラフトPR扱いで、R2R-on-wasm bring-up スタック統合前のため現在の main 上では CI 検証不可能
- wasm JIT（`clr.jit`）のコンパイルは確認済み
- R2R-on-wasm プロトタイプでの検証済み（修正前は crash、修正後は pass、R2R↔interpreter バッテリーは byte-identical）
- GitHub Copilot の支援下で作成

- **CoreCLRデータディスクリプタ追加**：RWメタデータ構造（StgPool、StgPoolSeg、MDInternalRW、CLiteWeightStgdbRW、CMiniMdRW、CMiniMdSchema）とEnCモジュール状態（EditAndContinueModule）の新規定義
- **マネージドコントラクトモデル拡張**：PEAssembly、Module、DataTypeに新フィールド/型を追加；Module.MetadataGenerationで動的メタデータの無効化追跡
- **メタデータキャッシング実装**：生成番号（RefEmit）またはEnC編集カウンタに基づく世代的キャッシング；前進実行時はキャッシュ保持、遡及実行時はフラッシュ
- **RWメタデータ再構成**：ストレージプール統合を含むECMA-335準拠メタデータイメージの生成
- **ドキュメント更新**：新メタデータソースと無効化シグナルを反映

#### パフォーマンスへの影響
提供データなし。ただし、メタデータキャッシング導入により前進実行中のメタデータ再読取が削減される可能性がある一方、RWメタデータ再構成の処理コストが発生します。
