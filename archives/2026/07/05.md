### [#130221](https://github.com/dotnet/runtime/pull/130221) Align keepnativesymbols option with src/tests/build {#130221}
- 作成者: [@am11](https://github.com/am11)
- 作成日時: 2026年07月05日 13:55:04(UTC)
- マージ日時: 2026年07月05日 16:55:20(UTC)
- ラベル: <a style="text-decoration:none;" href="../../labels/area-Infrastructure-coreclr/index.html"><span style="background-color: #fbca04; color: #000000; display: inline-block; padding: 0 7px; font-size:12px; font-weight:500; line-height:18px; border-radius:2em; border:1px solid transparent;">area-Infrastructure-coreclr</span></a> <a style="text-decoration:none;" href="../../labels/community-contribution/index.html"><span style="background-color: #c2e0c6; color: #000000; display: inline-block; padding: 0 7px; font-size:12px; font-weight:500; line-height:18px; border-radius:2em; border:1px solid transparent;">community-contribution</span></a>
#### 概要
ビルドスクリプト間の`--keepnativesymbols`オプションの動作を統一するPR。
従来、`./build.sh`ではこのオプションが値を必須とする引数形式（`--keepnativesymbols true`）でしたが、`src/tests/build.sh`ではスイッチ形式（値なし）で実装されていました。
本変更により、両スクリプトで統一してスイッチ形式（値なし）に変更されています。

#### 変更内容
- **eng/build.sh**: `--keepnativesymbols`オプションの実装をスイッチ形式に変更（値を不要にする）
- **docs/workflow/building/coreclr/android.md**: ドキュメントを更新して新しいスイッチ形式を反映

#### パフォーマンスへの影響
影響なし

#### 関連Issue
なし

#### その他
この変更はビルドスクリプトのインターフェース統一であり、ユーザーがビルドコマンドを実行する際の使用方法に影響します。既存のスクリプト呼び出しで`--keepnativesymbols true`という形式を使用していた場合は、更新後は`--keepnativesymbols`のみとなるため、自動化されたビルドスクリプトの更新が必要となる可能性があります。

### [#130213](https://github.com/dotnet/runtime/pull/130213) Publish GCHandle assignments with release semantics {#130213}
- 作成者: [@BradBarnich](https://github.com/BradBarnich)
- 作成日時: 2026年07月05日 02:06:27(UTC)
- マージ日時: 2026年07月05日 16:58:40(UTC)
- ラベル: <a style="text-decoration:none;" href="../../labels/arch-arm64/index.html"><span style="background-color: #eb6420; color: #ffffff; display: inline-block; padding: 0 7px; font-size:12px; font-weight:500; line-height:18px; border-radius:2em; border:1px solid transparent;">arch-arm64</span></a> <a style="text-decoration:none;" href="../../labels/area-GC-coreclr/index.html"><span style="background-color: #d4c5f9; color: #000000; display: inline-block; padding: 0 7px; font-size:12px; font-weight:500; line-height:18px; border-radius:2em; border:1px solid transparent;">area-GC-coreclr</span></a> <a style="text-decoration:none;" href="../../labels/community-contribution/index.html"><span style="background-color: #c2e0c6; color: #000000; display: inline-block; padding: 0 7px; font-size:12px; font-weight:500; line-height:18px; border-radius:2em; border:1px solid transparent;">community-contribution</span></a>
#### 概要
GCハンドル割り当て時にリリースセマンティクスを使用して、弱順序のARM64システムにおけるメモリ可視性の問題を修正します。従来のプレーンストア（`*handle = value`）では、オブジェクト初期化前の書き込みが他スレッドに可視化される前に、そのハンドルを通じてオブジェクト参照が公開される可能性がありました。

修正内容：`HndAssignHandle`と`HndAssignHandleGC`でプレーンストアを`VolatileStore`に変更し、ハンドル割り当てをリリース公開ポイントとします。

```cpp
// 修正前
*(_UNCHECKED_OBJECTREF *)handle = value;

// 修正後
VolatileStore((_UNCHECKED_OBJECTREF *)handle, value);
```

#### 変更内容
- **src/coreclr/gc/handletable.inl**: `HndAssignHandle`と`HndAssignHandleGC`内のGCハンドルスロットへの割り当てをプレーンストアからVolatileStoreに変更。メモリフェンスを挿入し、先行する書き込みがハンドルを通じたオブジェクト観測前に可視化されることを保証。

#### パフォーマンスへの影響
メモリ順序制約の厳格化により、ARM64システムにおいて若干のパフォーマンス影響がある可能性があります。ただし、本PR内に具体的なベンチマーク結果の記載はありません。

#### 関連Issue
[#130213](https://github.com/dotnet/runtime/issues/130213)

#### その他
- **検出背景**: ARM64実行時（10.0.9）のRegex.Replace実装で稀なNullReferenceExceptionが発生。WeakReferenceが部分初期化されたRegexReplacementオブジェクトを観測していることが原因。
- **検証**: WeakReference公開再現テストで検証。パッチなしではMacおよびAWS ARM64で即座に失敗、x64では失敗なし。パッチ適用後は両環境で失敗なし。
- **影響範囲**: ランタイムGCハンドル層の内部実装。公開APIへの直接的な変更なし。

### [#125379](https://github.com/dotnet/runtime/pull/125379) Optimize `string.Split(char, ...)` {#125379}
- 作成者: [@hamarb123](https://github.com/hamarb123)
- 作成日時: 2026年03月10日 12:26:29(UTC)
- マージ日時: 2026年07月05日 14:44:22(UTC)
- ラベル: <a style="text-decoration:none;" href="../../labels/area-System.Runtime/index.html"><span style="background-color: #d4c5f9; color: #000000; display: inline-block; padding: 0 7px; font-size:12px; font-weight:500; line-height:18px; border-radius:2em; border:1px solid transparent;">area-System.Runtime</span></a> <a style="text-decoration:none;" href="../../labels/tenet-performance/index.html"><span style="background-color: #c2e0c6; color: #000000; display: inline-block; padding: 0 7px; font-size:12px; font-weight:500; line-height:18px; border-radius:2em; border:1px solid transparent;">tenet-performance</span></a> <a style="text-decoration:none;" href="../../labels/community-contribution/index.html"><span style="background-color: #c2e0c6; color: #000000; display: inline-block; padding: 0 7px; font-size:12px; font-weight:500; line-height:18px; border-radius:2em; border:1px solid transparent;">community-contribution</span></a>
#### 概要
`string.Split(char, ...)` のパフォーマンスを最適化するPRです。単一文字セパレータ（最も一般的なケース）を対象とした特化した高速パスを追加し、ベクトル化実装によるセパレータ位置スキャンを導入しました。

#### 変更内容
- **String.Manipulation.cs**: `MakeSeparatorListAny` に単一セパレータ（`separators.Length == 1`）専用の高速パスを追加。新たに `MakeSeparatorListVectorized(..., char c)` オーバーロードを導入してセパレータスキャンをベクトル化。2～3セパレータパスは `separators.Length >= 2` を前提に調整。
- **SpanHelpers.Packed.cs**: `PackSources` ヘルパーを内部アクセスに変更し、複数の呼び出し元をサポート。
- **String.SplitTests.cs**: 新しい最適化パスに対するテストケースを追加（56行追加）。

#### パフォーマンスへの影響
単一文字セパレータで、セパレータがまれ/なしのケースが主な改善対象：
- **x64 (AMD)**: 約47%改善
- **ARM**: 約20%改善
- 単一文字のオーバーロードが文字列セパレータオーバーロードを上回るパフォーマンスを実現
- その他のケース（複数セパレータなど）は誤差範囲内での変化

.NET 12ではさらに約15%の改善余地が存在。

#### 関連Issue
[#125379](https://github.com/dotnet/runtime/pull/125379)

#### その他
- Copilotレビューで、`Vector512<byte> PackSources` ヘルパーのコメントが `Vector256<short>` を参照している点が指摘されており、メンテナンス性の観点から修正が推奨されています。
- `PackSources` ヘルパー使用時の制約条件（ISAサポート確認、`CanUsePackedIndexOf` 検証）のドキュメント化も提案されています。
