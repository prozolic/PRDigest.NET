#### 概要
WebAssembly（wasm）上のCoreCLR環境で、`libSystem.Globalization.Native`のP/Invoke EntryCountが不正な値に設定されていた問題を修正します。PR #130140で新しいP/Invoke関数が追加された際、EntryCountが33のままとなっており、テーブルの最後のエントリ（`GlobalizationNative_ToUnicode`）が未到達になっていました。これにより`IdnMapping.GetUnicode`/`ToUnicode`テストが`DllNotFoundException`で失敗していました。

#### 変更内容
- **src/coreclr/vm/wasm/browser/callhelpers-pinvoke.cpp**: `libSystem.Globalization.Native`のEntryCount を33 → 34に変更
- **src/coreclr/vm/wasm/wasi/callhelpers-pinvoke.cpp**: `libSystem.Globalization.Native`のEntryCount を33 → 34に変更

#### パフォーマンスへの影響
影響なし

#### 関連Issue
[#130479](https://github.com/dotnet/runtime/issues/130479)（blocking-clean-ci）、[#130140](https://github.com/dotnet/runtime/issues/130140)

#### その他
- 根本原因：PR #130140で生成ファイルを手動編集し、テーブルに新しいエントリを追加しましたが、hard-codedされたEntryCountの更新を忘れていました。`minipal_resolve_dllimport`は線形探索で`entries[0 .. count-1]`をスキャンするため、count=33では34番目のエントリ（`GlobalizationNative_ToUnicode`）が不可到達となり、P/Invokeがnullに解決されていました。
- `GlobalizationNative_GetAscii`（`ToAscii`、index 32）はカウント内にあるため、引き続きパスしていました。
- 通常、テーブル長からcountを生成するため、手動編集後は再生成すべきでした。

- 作成者: [@max-charlamb](https://github.com/max-charlamb)
- 作成日時: 2026年07月10日 19:52:13(UTC)
- マージ日時: 2026年07月11日 20:30:59(UTC)
- ラベル: <a style="text-decoration:none;" href="../../labels/area-Diagnostics-cdac/index.html"><span style="background-color: #d4c5f9; color: #000000; display: inline-block; padding: 0 7px; font-size:12px; font-weight:500; line-height:18px; border-radius:2em; border:1px solid transparent;">area-Diagnostics-cdac</span></a>
#### 概要
cDAC GC ストレステスト フレームワークにおけるプロセス終了時のクラッシュを修正します。arm/arm64 でのみ発生する `exit 139` (SIGSEGV) エラーが、検証完了後の teardown フェーズで発生していました。根本原因は、バックグラウンドスレッドがまだメモリを割り当てている最中に `CdacStressPolicy::Shutdown` が共有リソースを解放するという競合状態です。修正では、`Shutdown` 時に `s_cdacLock` を保持して `s_initialized` をクリアし、`VerifyAtStressPoint` が同ロック下で再チェックして撤去中の状態への検証を防ぎます。

#### 変更内容
- **src/coreclr/vm/cdacstress.cpp**: `Shutdown` で `s_cdacLock` を保持しながら `s_initialized` をクリアし、`VerifyAtStressPoint` 内でロック下での再チェックを追加。これにより、EE シャットダウン中のバックグラウンドスレッドによる使用後解放（use-after-release）を防止します。
- **eng/pipelines/runtime-diagnostics.yml**: パイプライン設定の軽微な更新。

#### パフォーマンスへの影響
影響なし。本修正は teardown ロジックの同期化に限定され、ストレステスト検証ロジック自体やランタイム動作に変更がありません。

#### 関連Issue
[#130447](https://github.com/dotnet/runtime/issues/130447) から分割

#### その他
- クラッシュは arm/arm64 環境のみで発生し、x64 では再現しません。x64 での 80 回連続実行テストでは、サマリ出力とともに正常に完了することを確認済みです。
- 本修正は公開 API ではなく、cDAC ストレステスト フレームワークの内部実装のみに影響します。

- ラベル: <a style="text-decoration:none;" href="../../labels/area-System.Net.Security/index.html"><span style="background-color: #d4c5f9; color: #000000; display: inline-block; padding: 0 7px; font-size:12px; font-weight:500; line-height:18px; border-radius:2em; border:1px solid transparent;">area-System.Net.Security</span></a> <a style="text-decoration:none;" href="../../labels/os-android/index.html"><span style="background-color: #fef2c0; color: #000000; display: inline-block; padding: 0 7px; font-size:12px; font-weight:500; line-height:18px; border-radius:2em; border:1px solid transparent;">os-android</span></a>
#### 概要
Android上の`SslStream`がプラットフォームの信頼インフラストラクチャ（`network_security_config.xml`を含む）を無視していた問題を修正します。`DotnetProxyTrustManager`を改修し、プラットフォームの`X509TrustManager`をラップするようにして、証明書ピニングやカスタム信頼アンカーが正しく機能するようになります。プラットフォームの検証結果と管理側の検証を組み合わせ、より厳格な評価を行います（プラットフォームが拒否した場合は管理側でもクリアできない）。

```csharp
// RemoteCertificateValidationCallback は両方の検証結果の合併集合を受け取るようになります
// プラットフォーム拒否 → RemoteCertificateChainErrors がプリセットされます
// プラットフォーム受理 → 管理側の検証が独立して実行され、エラーが追加される可能性があります
```

#### 変更内容
**Java層（DotnetProxyTrustManager）:**
- プラットフォームの`X509TrustManager`を`TrustManagerFactory`経由で取得しラップ
- `X509TrustManagerExtensions.checkServerTrusted`を使用してホスト名対応の検証を実施
- `chainTrustedByPlatform`フラグをネイティブコールバックに渡す

**ネイティブC層（pal_sslstream.c、pal_trust_manager.c、pal_jni.c）:**
- `SSLStreamCreate`を単一関数に統合（targetHost、trustCerts、keyManagersを受け取る）
- `GetX509TrustManager()`でプラットフォーム信頼マネージャーを返却、デフォルトインスタンスをJNIグローバル参照としてキャッシュ
- `VerifyRemoteCertificate`の登録時に戻り値をキャプチャしNativeAOT対応を強化
- 不要な`ApplyLegacyAndroidSNIWorkaround`と関連の削除
- `FreeSSLStream`の潜在的なSIGSEGV修正とJNIローカル参照リーク修正

**管理側C#（SslStream.Android.cs、SslStream.IO.cs等）:**
- `VerifyRemoteCertificate`が`ref SslPolicyErrors`を受け取るよう変更（呼び出し側が検証結果をプリセット可能）
- `GetTrustCertHandles`でカスタムルート証明書を収集
- `VerifyRemoteCertificate(bool chainTrustedByPlatform)`でプラットフォーム検証結果に基づいて`sslPolicyErrors`をプリセット

**ビルドインフラ（AndroidAppBuilder）:**
- `NetworkSecurityConfig`と`NetworkSecurityConfigResourcesDir`をサポート

#### パフォーマンスへの影響
プラットフォーム信頼マネージャーインスタンスをJNIグローバル参照としてキャッシュすることで、`TrustManagerFactory.init(null)`がハンドシェイク毎に呼ばれなくなります。これにより不要なJNI呼び出しが削減されます。
