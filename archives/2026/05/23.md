### [#128503](https://github.com/dotnet/runtime/pull/128503) \[cDAC\] Implement CheckContext for cDAC {#128503}
- 作成者: [@barosiak](https://github.com/barosiak)
- 作成日時: 2026年05月22日 21:20:12(UTC)
- マージ日時: 2026年05月23日 00:38:10(UTC)
- ラベル: <a style="text-decoration:none;" href="../../labels/area-Diagnostics-cdac/index.html"><span style="background-color: #d4c5f9; color: #000000; display: inline-block; padding: 0 7px; font-size:12px; font-weight:500; line-height:18px; border-radius:2em; border:1px solid transparent;">area-Diagnostics-cdac</span></a>
#### 概要
cDAC（Compact Data Access Components）の管理側実装に`CheckContext`機能を実装し、スタックポインタの範囲検証をサポートします。同時にネイティブDAC側から`DbgSkipStackCheck`設定による検証スキップ機能を削除し、セキュリティと一貫性を向上させます。

#### 変更内容
- **管理側cDAC実装（DacDbiImpl.cs）**: `CheckContext`メソッドを実装。コンテキストバッファをパースし、コントロールフラグが存在する場合、スレッドのキャッシュされたスタック境界に対してスタックポインタの範囲検証を実行
- **プラットフォームコンテキスト抽象化**: `IPlatformContext`と`IPlatformAgnosticContext`インターフェースに`ContextControlFlags`プロパティを追加。AMD64、ARM、ARM64、LoongArch64、RISC-V64、x86の6つのアーキテクチャ固有の実装で対応するフラグ値を提供
- **ネイティブDAC側（dacdbiimplstackwalk.cpp）**: `UNSUPPORTED_DbgSkipStackCheck`設定オプションと関連する不要なグローバル変数を削除。コントロールフラグが存在する場合は常にスタック範囲チェックを実行
- **HRESULT定義（CorDbHResults.cs）**: 管理側での整合性のため`CORDBG_E_NON_MATCHING_CONTEXT`定数を追加
- **ユニットテスト（DacDbiImplTests.cs）**: コントロールフラグあり/なしのシナリオと、全6アーキテクチャでのスタックポインタ範囲内・超過・境界値検証をカバーするテストを追加

#### パフォーマンスへの影響
影響なし。この変更はデバッグ時のスタック検証ロジックの実装場所を移行するもので、パフォーマンス特性には影響しません。

#### 関連Issue
なし

#### その他
- 破壊的変更：`DbgSkipStackCheck`設定オプションは削除されるため、この機能に依存していた場合は対応が必要です
- テストは「SP in-range、out-of-range、at-boundary検証」と「コントロールフラグなし時の早期終了でスタックデータが照会されないことの検証」をカバーしており、堅牢性が確保されています
- 内部実装（非公開API）の変更です

### [#128500](https://github.com/dotnet/runtime/pull/128500) JIT: fold isinst using reaching VNs and compareTypesForCast {#128500}

#### 概要
JITの全局アサーション伝播機能を強化し、PHIノードと値番号（VN）を活用して、常にtrueになる`isinst`/`chkcast`キャストをコンパイル時に折り畳むことを可能にしました。複数の分岐で異なる型のオブジェクトを生成していても、すべてが共通の基底クラスを継承している場合、冗長なキャストチェックを削除します。

**サンプルコード:**
```csharp
static void Test(int len)
{
    object bc;
    if (len > 100)
        bc = new DerivedClass1();
    else
        bc = new DerivedClass2();
    
    if (bc is BaseClass) // コンパイル後は常にtrueとして最適化される
        Console.WriteLine("bc is a BaseClass");
}
```

#### 変更内容
- **assertionprop.cpp**: `optAssertionIsSubtype`をリファクタリングして`bool`を返すように改善し、`optAssertionVNIsSubtype`を新規実装。VNベースのオブジェクト型情報とPHI到達分析を組み合わせてキャスト成功を証明
- **compiler.h/hpp**: アサーション伝播ヘルパー関数の署名を更新し、新しいVNベースサブタイプヘルパーを宣言
- **valuenum.cpp/h**: PHIノードからの型情報をVNレベルで復元する機能を追加
- **テスト**: `IsInstReachingVN.cs`にて新規最適化パターンのテストケースを追加

#### パフォーマンスへの影響
示例コード：
- **改善前**: 104バイトのコード生成（型チェック、分岐、HELP_ISINSTANCEOFCLASS呼び出し含む）
- **改善後**: 26バイトのコード生成（キャストチェック完全削除、常にtrueの分岐のみ残存）

約75%のコード削減とランタイムキャスト検証コストの除去による実行速度の向上が期待できます。

#### 関連Issue
なし

#### その他
- 内部実装の変更であり、公開APIへの影響はなし
- 互換性への破壊的変更なし
- PHI分析の深さに予算制限を設けることで、過度な解析オーバーヘッドを防止

### [#128499](https://github.com/dotnet/runtime/pull/128499) Restore E_NOTIMPL fallback in DacDbiInterfaceImpl::GetContext {#128499}
- 作成者: [@tommcdon](https://github.com/tommcdon)
- 作成日時: 2026年05月22日 18:37:02(UTC)
- マージ日時: 2026年05月23日 20:24:07(UTC)
- ラベル: <a style="text-decoration:none;" href="../../labels/area-Diagnostics-coreclr/index.html"><span style="background-color: #d4c5f9; color: #000000; display: inline-block; padding: 0 7px; font-size:12px; font-weight:500; line-height:18px; border-radius:2em; border:1px solid transparent;">area-Diagnostics-coreclr</span></a>
#### 概要
マネージドデバッグのスタックウォーク機能が非Windows platforms で正常に動作するよう、`DacDbiInterfaceImpl::GetContext` に `E_NOTIMPL` フォールバック機能を復元しました。`ICorDebugDataTarget::GetThreadContext` が `E_NOTIMPL` を返すプラットフォームで、Thread の Frame chain をウォークして REGDISPLAY から レジスタコンテキストを再構築します。

このフォールバックは以前の変更で削除されており、今回の修正によって復元されます。

#### 変更内容
- **src/coreclr/debug/daccess/dacdbiimpl.cpp**: `GetContext` メソッドの実装を更新
  - `GetThreadContext` が `E_NOTIMPL` を返した場合、エラーをスローせずフォールバック処理を実行
  - Thread の Frame chain をウォークして `UpdateRegDisplay` と `UpdateContextFromRegDisp` を使用し、マネージドスタックウォーク用のコンテキストを合成
  - `FEATURE_INTERPRETER` が定義されている場合、`InterpreterFrame` から `SetContextToInterpMethodContextFrame` を使用してコンテキストを合成する高速パス
  - 適切なマネージドフレームコンテキストを再構築できない場合、出力コンテキストをゼロクリア

#### パフォーマンスへの影響
影響なし（デバッグ実行時のフォールバック処理のため、パフォーマンスクリティカルパスではない）

#### 関連Issue
なし

#### その他
- DAC/DBI（Data Access Component / Debug Interface）に関連する内部実装の修正
- 非Windows platforms（Linux、macOS等）でのマネージドデバッグ機能の復旧
- Visual Studio内部テストを通じて検出された不具合の修正

#### 概要

#### 変更内容
- **src/coreclr/jit/importer.cpp**: tailcall-stress昇格時にintrinsic フラグをチェック追加（44行追加、30行削除）
- **src/coreclr/System.Private.CoreLib/src/System/Threading/Thread.CoreCLR.cs**: `Thread.ClearWaitSleepJoinState()`から一時的なworkaround属性（`[MethodImpl(NoInlining | NoOptimization)]`）を削除（2行削除）
- **テスト追加**: Runtime_122479向けの回帰テストを新規追加

#### パフォーマンスへの影響
影響なし（stress-tailcall処理のロジック調整のため、通常ビルドでの性能変化なし）

#### 関連Issue
[#122479](https://github.com/dotnet/runtime/issues/122479)、[#122652](https://github.com/dotnet/runtime/issues/122652)

#### その他
本修正により、`[SuppressGCTransition]`呼び出しを含むメソッドでの不正なBBJ_RETURNブロック生成が防止されます。importer.cppでのロジック修正により、workaroundアトリビュートの削除が可能になりました。
