### [#128048](https://github.com/dotnet/runtime/pull/128048) Limit sccache cache size to 3.5 GB to reduce transfer overhead {#128048}
- 作成者: [@agocke](https://github.com/agocke)

#### 概要
CoreCLR パイプラインの sccache ローカルキャッシュサイズを3.5GBに制限し、ローリングビルド時のキャッシュ復元・保存のオーバーヘッドを削減します。デフォルトの10GB制限では、古いエントリが蓄積して6～7GBまで増加し、キャッシュの転送に約3分のオーバーヘッドが発生していました。LRU削除により、作業セットサイズ（~3GB）に近い状態を維持します。

#### 変更内容
- **eng/pipelines/coreclr/templates/setup-sccache.yml**: `SCCACHE_CACHE_SIZE=3.5G` 環境変数を設定して、キャッシュサイズを制限し、古いエントリの自動削除を有効化。Azure Pipelines キャッシュのキーから `buildConfig` を除外し、PR ビルドがローリングビルドのキャッシュを活用できるように変更。
- **System.Security.Cryptography 関連ファイル**: `X25519DiffieHellmanCng.DeriveRawSecretAgreementCore` の部分オーバーライドに `unsafe` キーワードを追加（Windows実装、共通シム、非サポート実装）。

#### パフォーマンスへの影響
- **キャッシュサイズ**: 6～7GB（増加傾向）→ 3.5GB（安定化）
- **復元時間**: ~1分 → ~30秒
- **保存時間**: ~1分45秒 → ~50秒
- **総オーバーヘッド/レグ**: ~3分 → ~1.5分
- **レグあたりの実質的な時間短縮**: ~6分 → ~7.5分

キャッシュヒット率が低下した場合は制限値を引き上げることで調整可能。

#### 関連Issue
なし

#### その他
- Copilotによるレビューでは、PR説明とスコープの不一致を指摘。`buildConfig` キャッシュキー変更についての説明が不足していると指摘されている。
- System.Security.Cryptography の `unsafe` キーワード追加は、PR のスコープ（sccache キャッシュサイズ制限）と無関連な変更のため、分離すべき可能性がある。

### [#127950](https://github.com/dotnet/runtime/pull/127950) Fold more branches via O1K_VN \<relop\> O2K_VN comparisons {#127950}
- 作成者: [@EgorBo](https://github.com/EgorBo)
- 作成日時: 2026年05月08日 12:26:48(UTC)
- マージ日時: 2026年05月11日 17:02:13(UTC)
- ラベル: <a style="text-decoration:none;" href="../../labels/area-CodeGen-coreclr/index.html"><span style="background-color: #d4c5f9; color: #000000; display: inline-block; padding: 0 7px; font-size:12px; font-weight:500; line-height:18px; border-radius:2em; border:1px solid transparent;">area-CodeGen-coreclr</span></a>
#### 概要
CoreCLR JIT コンパイラの大域的アサーション伝播と範囲解析を拡張し、両方のオペランドが任意の値番号（VN）である符号付き比較（VN <relop> VN）に対応します。これにより、リレーショナルファクトをチェーン化することで追加的なブランチ折畳みが可能になります。

例えば `if (x > 10 && x < 100)` の後に `if (y > x)` がある場合、JIT は `y > 11` という推論を行い、その後の `return y > 0;` を常に真として最適化できます。

#### 変更内容
- **compiler.h**: 新しいグローバルアサーション演算子タイプ `O2K_VN` を追加して、"op2 が任意の値番号" を表現。VN間の関係演算アサーション用のファクトリメソッドを追加
- **assertionprop.cpp**: VN間の関係演算アサーションの生成ロジックを実装。JTRUE ノードで利益がある場合に VN間の関係演算アサーションを作成
- **rangecheck.h**: 境界付き再帰的な VN間アサーション解決をサポートする `MergeEdgeAssertionsWorker` メソッドを宣言
- **rangecheck.cpp**: エッジアサーション統合のワーカー/ラッパーパターンを実装。VN間の関係演算アサーションを使用した範囲の絞り込みロジックを追加

#### パフォーマンスへの影響
ブランチ折畳みの拡張により、複数の関連する条件チェックをコンパイル時に最適化できるようになります。これにより生成されるコードサイズが削減され、実行時パフォーマンスが向上する可能性があります。ただし、具体的なベンチマーク数値は提供されていません。

#### 関連Issue
なし

#### その他
この変更は JIT コンパイラの内部実装（範囲解析とアサーション伝播）に限定され、公開 API や破壊的な変更は含まれません。符号付き int32 の場合に限定して実装されています。
