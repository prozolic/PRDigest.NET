### [#127653](https://github.com/dotnet/runtime/pull/127653) Fix SIGBUS crash on ARMv7: use Unsafe.ReadUnaligned in Frozen.Hashing {#127653}
- 作成者: [@Copilot](https://github.com/apps/copilot-swe-agent)
- 作成日時: 2026年05月01日 16:24:03(UTC)

#### 概要
ARMv7（Android Unity IL2CPPなど）で`System.Collections.Frozen.Hashing`の`char*`から`uint*`へのキャスト時に発生するアンアライン32ビット読み取りによるSIGBUSクラッシュを修正します。`ToFrozenSet<string>(StringComparer.Ordinal)`を長さ≥4の文字列で呼び出す際の問題を解決します。

**修正内容：**
- `char*`から`uint*`への直接デリファレンスを`Unsafe.ReadUnaligned<uint>()`に置き換え、アンアライン読み取りを安全に処理
- 同じパターンが存在していた`GetHashCodeOrdinalIgnoreCaseAscii`にも適用

#### 変更内容
- **Hashing.cs** — `GetHashCodeOrdinal`と`GetHashCodeOrdinalIgnoreCaseAscii`内のアンアライン読み取り箇所で、`((uint*)src)[n]`と`uint* ptrUInt32`ループを`Unsafe.ReadUnaligned<uint>(ptr)`に置き換え
- `using System.Runtime.CompilerServices`を追加

#### パフォーマンスへの影響
`Unsafe.ReadUnaligned<uint>()`はx86/x64環境での直接デリファレンスと同等のパフォーマンスを持つため、影響なし（アンアライン読み取りが許容される環境では最適化される、許容されない環境では正しい動作を提供）

#### 関連Issue
なし

#### その他
この修正は内部実装（`System.Collections.Frozen`のハッシング機構）に限定される変更で、公開APIへの影響はありません。ARMv7などのアンアライン読み取りが制限される環境での実行時安定性向上が主な効果です。

### [#125795](https://github.com/dotnet/runtime/pull/125795) Include requesting assembly chain in assembly load failure exceptions {#125795}
- 作成者: [@Copilot](https://github.com/apps/copilot-swe-agent)
- 作成日時: 2026年03月19日 19:11:38(UTC)
- マージ日時: 2026年05月01日 16:07:27(UTC)
- ラベル: <span style="background-color: #d4c5f9; color: #000000; display: inline-block; padding: 0 7px; font-size:12px; font-weight:500; line-height:18px; border-radius:2em; border:1px solid transparent; cursor:default;">area-AssemblyLoading</span>
#### 概要
アセンブリロード失敗時の例外診断を改善し、依存関係の欠落や無効性の際に、ロード失敗をトリガーした要求元アセンブリのチェーンを追跡・表示するようにしました。これにより複雑な依存関係のシナリオでも開発者はロード失敗の原因を特定しやすくなります。

例：
```csharp
System.IO.FileNotFoundException: Could not load file or assembly 'LibC, Version=1.0.0.0, Culture=neutral, PublicKeyToken=null'. The system cannot find the file specified.
Requested by: LibB, Version=1.0.0.0, Culture=neutral, PublicKeyToken=null
 --> LibA, Version=1.0.0.0, Culture=neutral, PublicKeyToken=null
 --> helloworld, Version=1.0.0.0, Culture=neutral, PublicKeyToken=null
```

#### 変更内容
**マネージド例外クラス（公開API）**
- `BadImageFormatException`、`FileLoadException`、`FileNotFoundException`に`_requestingAssemblyChain`フィールドを追加
- `ToString()`メソッドを拡張し、"Requested by: ..."セクションを表示
- 新しいローカライズ対応リソース文字列`IO_FileLoad_RequestedBy`を追加

**ランタイム/アセンブリバインディング（内部実装）**
- CoreCLRの`AssemblySpecBindingCache`から親（要求元）アセンブリをクエリできる新メソッドを実装
- アセンブリバインディングチェーンを深さ10まで走査してチェーンを構築
- `EEFileLoadException`を更新し、要求元アセンブリチェーンをキャプチャ・伝播
- アンマネージド→マネージド例外生成パスを更新し、チェーンをマネージド例外コンストラクタに渡す
- `appdomain.cpp`/`hpp`と`assemblyspec.cpp`/`hpp`にヘルパー関数を追加（`FindCachedParentAssembly`等）

**テスト**
- 3つのテストアセンブリ（`MissingDependency.Root`、`MissingDependency.Mid`、`MissingDependency.Leaf`）を追加
- 推移的依存関係が欠落した際に例外メッセージにチェーンが表示されることを検証

#### パフォーマンスへの影響
ベンチマーク結果なし。ロード失敗時のみチェーン構築処理が動作するため、通常のロード成功時には影響なし。チェーン構築はバインディングキャッシュの反復処理で、深さ制限は10に設定されている。

#### 関連Issue
[#9185](https://github.com/dotnet/runtime/issues/9185)

#### その他
チェーンは最良努力ベース。アセンブリが複数の異なる親により要求される場合、イテレーション中に見つかった最初にキャッシュされたアセンブリの親を使用。CoreCLRコンパイラまたはランタイムの変更ですが、マネージド例外の`ToString()`出力のみに影響し、破壊的変更ではありません（新パラメータは内部パスでのみ使用）。
